Key engineering achievements

  • ANSI-safe date parsing — discovered that Databricks Serverless enforces ANSI SQL mode, where to_date() throws on bad values rather than returning NULL. Fixed with a reusable parse_date() helper using try_to_date() via F.expr().
  • Serverless Workflow design — built the 4-task DAG without job_clusters — a Free Edition constraint that required debugging the REST API response. Serverless compute is implicit.
  • Ambiguous join resolution — pre-dropped shared columns (retailer_id, customer_id) from the orders table before joining with parcels, resolving a Spark AMBIGUOUS_REFERENCE error.
  • Class-imbalanced ML — combined scale_pos_weight (~14×) with compute_sample_weight('balanced') and a decision threshold of 0.4 to maximise recall on the ~6% RTS minority class.
  • API-first deployment — all 4 notebooks uploaded and a Workflow job created entirely via Databricks REST API. No UI clicks required — fully reproducible.

Databricks satisfies all five in a single platform.

💡
Key insight

Databricks' greatest advantage is unification. Rather than stitching together separate tools for ETL, ML training, experiment tracking, and orchestration, all four run natively on the same compute fabric. No data movement between systems, no cross-tool auth to manage, no version skew between the transformation layer and the ML layer.

Why not dbt?

dbt excels at SQL-based transformations with version-controlled models and lineage tracking. But this project required complex PySpark operations (create_map(), Window, F.expr()), MLflow logging tightly coupled to training code, and a Pandas-backed ML step that runs in the same environment as the cleansing step. dbt would handle the Gold aggregation layer — but nothing else. Pure PySpark on Databricks was the cohesive choice.

⚠️
Real-world constraint discovered

Databricks Free Edition (AWS) only supports serverless compute. Workflow job definitions cannot include a job_clusters key — the API returns HTTP 400. Traditional cluster specs (node_type_id, num_workers) are also invalid. This was discovered and debugged live — the fix was removing all cluster configuration and relying on the implicit serverless runtime.

Chapter 03Part II — Pipeline
Medallion Architecture
Why three layers? What the contract is at each boundary, and how Delta Lake makes it reliable

The Medallion Architecture (multi-hop pattern) organises data into progressive quality layers — each with a clear contract, a defined consumer, and independent restartability.

The three-layer data contract

BRONZEpargo_bronzeOFF — accepts anythingSilver cleansing job onlyAppend (audit trail)
SILVERpargo_silverON — typed, enforcedGold aggregations, ML feature eng.Overwrite with dedup
GOLDpargo_goldON — typed, aggregatedDashboards, reports, analysts, ML scoringFull overwrite (idempotent)

Why not write clean data directly?

The most common ETL mistake is combining ingestion and cleansing. This creates three problems:

  1. No audit trail — if a cleaning rule is wrong, you cannot trace back to the original value
  2. Brittle pipelines — a new dirty pattern breaks the entire pipeline instead of quarantining bad rows
  3. No partial reruns — if Gold fails after Silver succeeds, you must rerun from the beginning

ℹ️
Open Standard

Delta Lake is Apache 2.0 licensed, maintained by the Linux Foundation. Data written by Databricks can be read by any engine that supports the Delta protocol — providing portability that proprietary formats like Snowflake's internal format do not offer.

Chapter 04
Bronze Layer
Raw ingestion — permissive by design. 52,000 rows with realistic enterprise dirt.

The Bronze layer is deliberately schema-permissive. Every column is STRING. Negative weights land as-is. Duplicates land as-is. Mixed-case statuses, five different date formats, garbage booleans — all land as-is. This is not a bug. It is the feature that makes the pipeline auditable and restartable.

⚠️
Design Decision

Casting columns to typed values at ingestion time means a single bad value rejects the entire row — or silently corrupts data. By landing everything as STRING in Bronze, 100% row retention at ingestion is guaranteed. The Silver layer then applies explicit, auditable casts with fallback logic. Bad values become DQ flags rather than lost records.

Chapter 05
Silver Layer
Deduplication, type casting, status normalisation — and the ANSI mode production bug

Silver is the most technically complex layer. Raw STRING data from Bronze becomes strongly-typed, deduplicated, canonically-normalised records that downstream consumers can trust without further validation.

1. Deduplication with Window functions

Duplicate detection uses ROW_NUMBER() over a window partitioned by the business key, ordered by ingestion timestamp descending — keeping the most recently loaded version:

02_silver_cleansing.py — deduplication
from pyspark.sql.window import Window

w = Window.partitionBy('parcel_id').orderBy(F.col('load_ts').desc())
deduped = (
    raw_parcels
    .withColumn('rn', F.row_number().over(w))
    .filter('rn = 1')          # keep only the latest record
    .drop('rn')
)
# Result: 52,000 → 49,999 rows  (2,001 duplicates removed)

2. Status normalisation with create_map()

02_silver_cleansing.py — canonical status map
status_map = {
    'DELIVERED':'DELIVERED', 'delivered':'DELIVERED', 'DELVRD':'DELIVERED',
    'IN_TRANSIT':'IN_TRANSIT', 'In Transit':'IN_TRANSIT',
    'RTS':'RTS', 'Return to Sender':'RTS', 'RETURN_TO_SENDER':'RTS',
    # ... 15 more variant mappings
}
map_expr = F.create_map([F.lit(x) for pair in status_map.items() for x in pair])
# Lookup map: O(1) per row — no CASE WHEN chain
status_clean = F.coalesce(map_expr[F.col('status')], F.lit('UNKNOWN'))

3. The ANSI mode bug — discovered and fixed

🐛
Production Bug Found

Databricks Serverless enforces ANSI SQL mode by default. In ANSI mode, F.to_date(col, 'yyyyMMdd') does not return NULL on bad values — it throws a DateTimeException, aborting the entire job. Standard Spark behaviour returns NULL. The fix: replace all to_date() with try_to_date() — a SQL function not available as a native PySpark function, so called via F.expr() with backtick-quoted column names.

02_silver_cleansing.py — ANSI-safe date parser (5 formats)
def parse_date(col_name):
    """try_to_date() returns NULL on parse failure — required for ANSI mode"""
    return F.coalesce(
        F.expr(f"try_to_date(`{col_name}`, 'yyyy-MM-dd')"),   # 2024-08-11
        F.expr(f"try_to_date(`{col_name}`, 'dd/MM/yyyy')"),   # 11/08/2024
        F.expr(f"try_to_date(`{col_name}`, 'dd-MM-yyyy')"),   # 11-08-2024
        F.expr(f"try_to_date(`{col_name}`, 'yyyyMMdd')"),     # 20240811
        F.expr(f"try_to_date(`{col_name}`, 'dd MMM yyyy')")   # 11 Aug 2024
    )
# NULL on all-formats failure → dq_date_unparsed flag raised

4. DQ flags — bad records stay visible

Rather than discarding unresolvable values, Silver attaches boolean DQ flags so downstream analysts can filter or audit:

  • dq_weight_null — weight was negative (clamped to NULL) or originally missing
  • dq_date_unparsed — created_date failed all 5 format patterns
  • dq_province_null — province was blank or missing (31.2% of parcels)

The Gold join bug — and the fix

🐛
AMBIGUOUS_REFERENCE Error

Joining pargo_silver.parcels and pargo_silver.orders on parcel_id caused a Spark AMBIGUOUS_REFERENCE error — both tables had retailer_id and customer_id columns. Fix: explicitly drop shared columns from the orders table before the join.

03_gold_analytics.py — pre-drop fix
# Drop shared columns from orders BEFORE join — keeps parcels version
_orders_join = orders.drop('retailer_id', 'customer_id')
parcels_with_order = parcels.join(_orders_join, on='parcel_id', how='left')
# Now retailer_id unambiguously refers to parcels.retailer_id

Class imbalance — three-pronged approach

04_ml_rts_prediction.py — imbalance handling
# 1. Sample weights — upweight minority class during fit
sample_weights = compute_sample_weight('balanced', y_train)

# 2. scale_pos_weight — ~14x on RTS class in XGBoost loss
model = XGBClassifier(
    scale_pos_weight = (y_train == 0).sum() / (y_train == 1).sum(),
    n_estimators=300, max_depth=5, learning_rate=0.05,
    subsample=0.8, early_stopping_rounds=20,
)
model.fit(X_train, y_train, sample_weight=sample_weights, ...)

# 3. Lower prediction threshold: 0.5 → 0.4
#    Catches more true RTS at cost of more false positives
y_pred = (y_pred_proba >= 0.4).astype(int)

💡
Why AUC 0.81 is meaningful

A random classifier scores AUC 0.50. AUC 0.81 means the model correctly ranks a random RTS parcel above a random non-RTS parcel 81% of the time. For an intervention use case, recall matters more than precision — catching 68% of at-risk parcels early generates significantly more value than a high-precision model that catches very few.
