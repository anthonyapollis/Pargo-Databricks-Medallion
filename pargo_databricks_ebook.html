Executive Summary
What was built, why it matters, and what a hiring manager should take away in 5 minutes

Pargo Parcels is South Africa's leading parcel locker and pickup point network, enabling e-commerce retailers to offer affordable last-mile delivery at over 4,000 pickup locations. The business generates high volumes of parcel events daily — and suffers a measurable cost each time a parcel is returned to sender (RTS): re-handling, re-delivery scheduling, and customer friction.

What was built

This project delivers a fully automated, serverless data pipeline on Databricks that takes raw, dirty parcel data through four progressive quality layers — and ends with a machine learning model that predicts, before dispatch, which parcels are at risk of being returned.

Why Databricks?
Platform selection rationale — and how this compares to Snowflake, AWS Glue, and dbt

When selecting a platform for a parcel analytics pipeline with an ML component, the core requirements were: unified compute for both transformations and ML training, a storage format with ACID guarantees and time travel, built-in experiment tracking, serverless execution (cost savings), and a familiar Python-first developer experience.

Key insight

Databricks' greatest advantage is unification. Rather than stitching together separate tools for ETL, ML training, experiment tracking, and orchestration, Databricks provides all four natively on the same compute fabric. This reduces operational complexity significantly — there is no data movement between systems, no cross-tool authentication to manage, and no version skew between the transformation layer and the ML layer.

Why not dbt?

dbt excels at SQL-based transformations with version-controlled models and lineage tracking. It is the right tool when your team works primarily in SQL and needs documented data models with tests. However, this project required:

  • Complex PySpark transformations (window functions, create_map(), F.expr()) not expressible in SQL
  • Pandas-backed ML training that runs in the same environment as the cleansing step
  • MLflow logging and model registration tightly coupled to the training code

dbt could handle the Gold aggregation layer well, but would require a Python dbt adapter and would not handle the Bronze or ML layers at all — making it a partial solution requiring additional tooling. For Pargo's use case, pure PySpark + Databricks Workflows was the more cohesive choice.

Databricks Free Edition constraints navigated

⚠️
Real engineering decision — not a tutorial

The Databricks Free Edition on AWS only supports serverless compute. This means Workflow job definitions cannot include a job_clusters key — the API returns HTTP 400 if you do. Traditional compute specs (node_type_id, num_workers) are also not valid. This was discovered and debugged during this project — the fix was to remove all cluster configuration from the job body and rely on the implicit serverless runtime.

Chapter 3
Medallion Architecture
Why three layers? What the contract is at each boundary, and how Delta Lake makes it work

The Medallion Architecture (or Multi-Hop architecture) is a data design pattern popularised by Databricks that organises data into progressive quality layers, each named after a precious metal to indicate increasing refinement.

The three-layer data contract

LayerDatabaseSchema enforcementWho reads itImmutability
BRONZEpargo_bronzeOFF — accepts anythingSilver cleansing job onlyAppend-only (historical audit)
SILVERpargo_silverON — enforced typesGold aggregations, ML feature engineeringOverwrite with dedup
GOLDpargo_goldON — typed & aggregatedDashboards, reports, analysts, ML scoringFull overwrite (idempotent)

Why not just write clean data directly?

The most common mistake in ETL is combining ingestion and cleansing into a single step. This creates three problems:

  1. No audit trail — if a cleaning rule is wrong, you cannot trace back to the original value
  2. Brittle pipelines — a new dirty pattern in source data breaks the entire pipeline instead of quarantining the bad rows
  3. No partial reruns — if the Gold step fails after Silver succeeds, you must rerun from the beginning

The medallion pattern solves all three. Bronze is the raw archive. Silver is the clean canonical representation. Gold is the business view. Each layer can fail, be reprocessed, or be modified independently.

Why Delta Lake?

Delta Lake is the storage format used across all three layers. It adds four critical capabilities over plain Parquet:

FeatureWhat it enables in this project
ACID transactionsConcurrent reads/writes to Bronze won't corrupt the table even if a job fails mid-write
Schema enforcementSilver and Gold reject rows that don't match the defined schema, catching upstream issues early
Time travelSELECT * FROM pargo_silver.parcels VERSION AS OF 3 — audit any prior state
MERGE / upsertIncremental Silver updates can merge new records without full rewrites
Z-Ordering / optimizeGold tables can be compacted and co-located on created_date for fast dashboard queries
ℹ️
Open table format

Delta Lake is an open-source format (Apache 2.0 licence) maintained by the Linux Foundation. The same Delta tables written by Databricks can be read by Trino, Apache Spark, Flink, DuckDB, and more — providing portability that proprietary formats like Snowflake's internal format do not.

Bronze Layer — Raw Ingestion
Why permissive ingestion is a feature, not a bug — and what "dirty data" looks like in production

The Bronze layer is deliberately schema-permissive. Every column is STRING. Negative weights land as-is. Duplicate rows land as-is. Mixed-case statuses land as-is. This is intentional.

⚠️
Design decision: why STRING for everything?

Casting a column to INTEGER or DATE at ingestion time means a single bad value rejects the entire row — or, worse, silently truncates data. By storing everything as STRING in Bronze, you guarantee 100% row retention at ingestion. The Silver layer then applies explicit, auditable casts with fallback logic. Bad values become DQ flags rather than lost records.

What Bronze does NOT do

  • Does not validate values — a status of XYZABC is accepted
  • Does not deduplicate — duplicate parcel IDs land as separate rows
  • Does not enforce referential integrity — unknown courier IDs are accepted
  • Does not parse dates — 20240811 and 2024-08-11 land side by side as text
Chapter 5
Silver Layer — Cleansing & Validation
Deduplication, type casting, status normalisation, and the ANSI mode gotcha

Silver is the most technically complex layer. It receives raw STRING data from Bronze and produces strongly-typed, deduplicated, canonically-normalised records that downstream consumers can trust without further validation.

1. Deduplication with Window functions

Duplicate detection uses ROW_NUMBER() over a window partitioned by the business key, ordered by ingestion timestamp descending — keeping the most recently loaded version of any duplicated record:

02_silver_cleansing.py — deduplication
from pyspark.sql.window import Window

w = Window.partitionBy('parcel_id').orderBy(F.col('load_ts').desc())
deduped = (
    raw_parcels
    .withColumn('rn', F.row_number().over(w))
    .filter('rn = 1')          # keep only the latest
    .drop('rn')
)
# Result: 52,000 → 49,999 rows (2,001 duplicates removed)

2. Status normalisation with create_map()

The status column arrives in 20+ variants. Rather than using a long CASE WHEN chain, a PySpark column map is created and applied as a lookup:

02_silver_cleansing.py — status mapping
status_map = {
    'DELIVERED':'DELIVERED', 'delivered':'DELIVERED', 'DELVRD':'DELIVERED',
    'IN_TRANSIT':'IN_TRANSIT', 'In Transit':'IN_TRANSIT', 'IN TRANSIT':'IN_TRANSIT',
    'RTS':'RTS', 'Return to Sender':'RTS', 'RETURN_TO_SENDER':'RTS',
    # ... 15 more mappings
}
map_expr = F.create_map([F.lit(x) for pair in status_map.items() for x in pair])
status_clean = F.coalesce(map_expr[F.col('status')], F.lit('UNKNOWN'))

3. ANSI mode — the critical production bug

🐛
Bug discovered and fixed

Databricks Serverless enforces ANSI SQL mode by default. In ANSI mode, F.to_date(col, 'yyyyMMdd') does not return NULL on bad values — it throws a DateTimeException, aborting the entire job. This is different from standard Spark behaviour where to_date() silently returns NULL. The fix was to replace all to_date() calls with try_to_date() — a SQL function that returns NULL on parse failure. Since try_to_date() is not available as a native PySpark function, it must be called via F.expr().

02_silver_cleansing.py — ANSI-safe date parsing helper
def parse_date(col_name):
    """
    Tolerates unparseable dates by returning NULL.
    Required because Databricks Serverless enforces ANSI mode,
    where to_date() throws on bad values instead of returning NULL.
    """
    return F.coalesce(
        F.expr(f"try_to_date(`{col_name}`, 'yyyy-MM-dd')"),
        F.expr(f"try_to_date(`{col_name}`, 'dd/MM/yyyy')"),
        F.expr(f"try_to_date(`{col_name}`, 'dd-MM-yyyy')"),
        F.expr(f"try_to_date(`{col_name}`, 'yyyyMMdd')"),
        F.expr(f"try_to_date(`{col_name}`, 'dd MMM yyyy')")
    )

# Usage:
.withColumn('created_date', parse_date('created_date'))
# 'created_date' tries 5 formats in order — NULL on all failures

The Gold join bug — and how it was fixed

🐛
AMBIGUOUS_REFERENCE error

When joining pargo_silver.parcels and pargo_silver.orders on parcel_id, both tables contained a retailer_id and customer_id column. After the join, any reference to retailer_id caused Spark to throw AMBIGUOUS_REFERENCE — it could not determine which table's column was intended.

Fix: explicitly drop the shared columns from the orders table before the join. This keeps the parcels version of those columns and avoids ambiguity entirely.

03_gold_analytics.py — resolving column ambiguity
# Both parcels and orders have retailer_id, customer_id
# Drop from orders BEFORE joining — keeps the parcels version
_orders_join = orders.drop('retailer_id', 'customer_id')

parcels_with_order = parcels.join(_orders_join, on='parcel_id', how='left')
# Now retailer_id unambiguously refers to parcels.retailer_id

Class imbalance

The target class (RTS = TRUE) represents only ~6.2% of parcels — a 14:1 imbalance. Without correction, a model predicting "Not RTS" for every parcel would achieve 94% accuracy while being completely useless. Two techniques were combined:

04_ml_rts_prediction.py — class imbalance handling
from sklearn.utils.class_weight import compute_sample_weight

# 1. Sample weights during training (sklearn)
sample_weights = compute_sample_weight('balanced', y_train)

# 2. XGBoost scale_pos_weight — upweights positive class in loss
model = XGBClassifier(
    scale_pos_weight = (y_train == 0).sum() / (y_train == 1).sum(),
    # ~14x weight on RTS class
    ...
)
model.fit(X_train, y_train, sample_weight=sample_weights, ...)

# 3. Lower prediction threshold (0.5 → 0.4)
# Catches more true RTS at cost of more false positives
y_pred = (y_pred_proba >= 0.4).astype(int)

Why AUC 0.81 is meaningful

A random classifier scores AUC 0.50. An AUC of 0.81 means the model correctly ranks a randomly chosen RTS parcel above a randomly chosen non-RTS parcel 81% of the time. For an intervention-based use case like this, recall matters more than precision — catching 68% of RTS parcels early (even with some false positives) generates significantly more value than a high-precision, low-recall model that catches very few.

Chapter 8
Databricks Workflows — Automation
4-task DAG, serverless compute, CRON scheduling, and the API-first deployment

All four notebooks are wired into a Databricks Workflow — a managed DAG orchestrator built into the platform. The Workflow runs daily at 03:00 SAST, uses serverless compute (zero cold start), and enforces task dependencies so a Gold failure never leaves Silver data inconsistent.

Workflow definition

Workflow DAG — task dependency chain
bronze_ingestion
    → silver_cleansing          (depends_on: bronze_ingestion)
        → gold_analytics        (depends_on: silver_cleansing)
            → ml_prediction     (depends_on: gold_analytics)

Schedule: "0 0 3 * * ?"        # Quartz CRON — 03:00 every day
Timezone: "Africa/Johannesburg"    # SAST
Compute:  Serverless              # no job_clusters key — Free Edition requirement
Status:   PAUSED                  # ready to unpause for production

API-first deployment

Notebooks were deployed to the Databricks workspace programmatically via the REST API — not by clicking in the UI. This enables reproducible, version-controlled deployments:

upload_to_databricks.py — REST API deployment
# 1. Upload notebook as base64-encoded Python source
api("/api/2.0/workspace/import", {
    "path":    "/Pargo_Medallion/01_bronze_ingestion",
    "format":  "SOURCE",          # native .py format
    "language":"PYTHON",
    "content": base64.b64encode(source.encode()).decode(),
    "overwrite":True
})

# 2. Create Workflow job
api("/api/2.1/jobs/create", {
    "name": "Pargo Medallion Pipeline",
    "schedule": {"quartz_cron_expression": "0 0 3 * * ?", ...},
    "tasks": [{
        "task_key": "bronze_ingestion",
        "notebook_task": {"notebook_path": "/Pargo_Medallion/01_bronze_ingestion"}
        # NO job_clusters key — serverless is implicit
    }, ...]
})
ℹ️
Serverless compute savings

In Databricks Free Edition (and in paid tier serverless jobs), compute is provisioned on-demand and billed per DBU-second with no idle time. A traditional job cluster starts in 5–8 minutes and bills from the moment the cluster is requested. For a daily pipeline that runs for ~15 minutes, serverless eliminates 8+ minutes of idle billing per run — a meaningful cost saving at scale.

  1. Parcels with rts_probability >= 0.4 are flagged as rts_predicted = 1
  2. A notification (email/SMS) can be triggered to the retailer or customer
  3. A Pargo agent can proactively contact the pickup point to increase collection window
  4. High-risk retailers can receive additional collection reminders automatically

This pipeline converts raw dirty operational data into a proactive risk management system. Before this, RTS was discovered after the fact. With this model, at-risk parcels are flagged 24–48 hours before the collection deadline — giving Pargo time to intervene and prevent the return entirely.
